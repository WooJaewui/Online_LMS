
 소스 코드 이관 - SW 배포.



 # 주요 용어 정의.

용어.
소스 코드 이관 : 공유되어야 할 소스 코드를 지정된 시스템에 등록하는 행위.
형상 관리 : SW에 대한 소스 코드의 등록, 변경, Release 등을 종합적으로 관리하는 활동.
Commit : 현 개발자 시스템에 있는 소스 코드의 변경사항을 형상관리 시스템에 반영.
Upload : 공유 / 등록 대상 소스 코드를 형상관리 시스템에 최초 등록.
umask : 운영체제에 존재하는 파일의 사용자 / 그룹 접근 허용 여부를 나타내는 체계.



 # 소스 코드 이관 개요.

소스 코드 이관.
공유되어야 할 소스 코드를 지정된 시스템에 등록하는 행위.
공유 대상 소스 코드를 등록 후, 협업 개발자가 공동으로 활용.

개발 시스템 활용 현황.
단독 개발 : 소스 코드 이관을 생각하지 않아도 된다.
공동 개발 : 형상 관리 시스템에 공유 해야 할 소스 코드를 업로드 해서 사용.

소스 코드 이관의 역할.
소스 코드 공유 : 필수 공유 소스 코드의 공유.
Baseline 설정 : 소스 코드 관리 시작 시점 수립( Revision 관리의 시작 ) 
협업 시작 : 이전까지는 개별 개발, 이관 이후부터 협업.
공동 책임 : 개발 SW에 대한 공동 책임.



 # Commit, Upload.

소스 코드 이관 체계.
이관 대상 : Header File, Include File, Function File, Config File 등.
이관 시점 : 개발 초기, 개발 중 / 후기. ( 개발 기간 중 필요하면 언제 든지 )
이관 방법 : 소스 코드 공유 방법.

Upload.
소스 코드를 형상 관리 시스템에 최초 등록.

Upload 순서.
소스 코드 선정 - > Upload 위치 선택 - > Log 입력 - > Upload.

Commit.
소스 코드의 변경 사항을 형상 관리 시스템에 반영.

Commit 순서.
대상 소스 코드 선정 - > Log 입력 - > Commit.



 # Option - Flag, umask 설정.

용어.
Option : 특정 기능의 동작 범위를 선택, 동작 방식의 선택.
Flag : 특정 기능의 식별자, 기능의 사용 여부.
umask : 운영체제에 존재하는 파일의 사용자 - 그룹 접근 허용 여부를 나타내는 체계.

활용 배경.
운용체제 종류  : 시스템의 운영체제의 종류에 따라 다르게 설치.
Libaray Version : Library Version에 따라 각기 다른 Component 설치.
SW Checking : 사용자 시스템에 존재하는 특정 SW의 존재 여부 확인.
사용자 접근 제한 : 특정 파일에 대해 User의 접근을 사전 방지.

umask 설정.
RWX : Read, Write, eXecute.
Read 0--.
Write : -0-.
eXecute : --0.
사용자 : 000 --- ---.
사용자 그룹 : --- 000 ---.
Ext ( 기타 ) : --- --- 000.
