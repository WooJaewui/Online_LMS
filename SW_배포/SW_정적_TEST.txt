 # SW Test 개요.

SW Test 정의.
결함( fault )을 찾기 위해 SW를 작동시키는 일련의 행위와 절차.
배포 환경에 따라 다양한 기법 적용.

SW Test 실행 목적.
SW가 원하는 목표치까지 완성되었는지 검증을 하기 위해서 실행.

품질 보증.
SW Test를 통해 SW가 원하는 목표치까지 달성하였을 때 품질 보증이 되었다고 할 수 있다.

SW Test 수행 조직.
개발자 : 별도의 Test 담당자가 없을 때 시행. ( 효율성이 높고, 독립성이 낮다 )
Tester : Test만을 위한 전문가. ( 효율성이 낮고, 독립성이 높다 )
QA : 개발 초기부터 제품 배포까지 전 과정에 참여. ( 효율성이 가장 낮고, 독립성이 가장 높다 )

SW Test 기법.
화이트 박스 Test : 프로그램 내부 로직을 보면서 Test. ( 구조 Test )
블랙 박스 Test : 프로그램 외부 명세를 보면서 Test. ( 기능 Test - 결과 값을 확인 )
동적 Test : 프로그램 실행을 요구하는 Test.
정적 Test : 프로그램 실행 없이 구조를 분석하여 논리성 검증.
검증( Validation ) : 과정을 Test. 사용자 중심. ( 사용자의 요구를 확인 ) 
확인( Verification ) : 결과를 Test, 개발자 중심. ( 개발자가 생각한 대로 실행되는지 확인 )



 # SW 정적 Test 개요.

SW 정적 Test 정의.
프로그램 실행 없이 구조 및 과정 등을 분석하여 검증하는 방법.
요구사항의 준수 여부, 예외 처리, 여타 Module과의 연관성 등의 확인.

SW 정적 Test 시행 환경.
개발 Project 진행  ->  Component 개발 & 취합  ->  SW Test  -> SW Package.
개발 Project 진행 과정에서도 SW Test를 할 수 있다. ( 자기 자신이 하는 경우가 많다 )

SW 정적 Test 사례.
Bug 수정.
필수적인 예외처리의 누락 .
사용자가 요구한 기능의 미구현.
성능 저하.
확장성, 유연성 부족.
개발 규칙 미준수.

SW 정적 Test 시행 절차.
계획 수립 : 정적 Test 일정, 범위, 수행자, 대상 결정.
범위 설정 : 목적, 절차, 개발 내역, 설명.
점검 : 소스코드 점검 시행 질의 / 응답.
수정 : 지적 사항 수정.
반영 : SW Packaging.



 # SW 정적 Test 실무.

SW 정적 Test 기법.
Inspection : 격식을 갖춘 정적 Test방법, 정해진 장소에 미리 선정된 인원참가, 내용과 대상 등 사전 공지.
Work Through : 개발 내역을 공유 & 설명, 문제점 발견 & 공동 해결 수행. ( 여러 명이 같이 해결 )
Peer Review : 일반적으로 1:1 진행, 개발 사항 설명 , 개발 Logic의 타당성 검토. ( 1명의 동료가 지원 )

SW 정적 Test 범위.
기능 구현 : 사용자가 요구한 기능의 구현 여부 확인.
Bug 원인 발견 : 개발자가 인지한 Bug의 원인 규명.
Logic 점검 : 목표한 기능 구현의 적절성 확인( 기능성, 비기능성 등 )
예외처리 적용 : 개발자가 인지하지 못한 예외처리 대상의 발견.
타당성 : 개발 순서, 일정, 도구 등의 개발 체계의 적절성 평가.
